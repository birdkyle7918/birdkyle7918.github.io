🔵 MySQL 的内部架构

分析器：相当于编译功能。

优化器：对执行过程进行优化，决定走索引或者不走索引之类的。

🔵 MySQL 的日志

error log

错误日志。默认开启的日志，而且不能被关闭，其中记录了：服务器启动和关闭过程中的信息（未必是错误信息，如 MySQL 如何启动 InnoDB 的表空间文件的、如何初始化自己的存储引擎的等等）、服务器运行过程中的错误信息、事件调度器运行一个事件时产生的信息、在从服务器上启动服务器进程时产生的信息。

bin log

二进制日志。该日志内容可以认为就是执行过的 SQL 语句脚本，这些语句可以再次执行。

在主从复制中，从库利用主库上的 bin log 进行脚本重播，从而实现主从同步。虽然开启该日志后对性能有所影响，但是影响不大，建议开启。该日志放置的位置在 MySQL 配置文件中有给出。

relay log

当要使用主从复制的时候，就要使用中继日志了，它是从 master 数据库上记录下来的二进制日志文件，再同步到自己的数据库上的中继日志文件，然后读取中继日志文件当中的内容来实现最终的同步。

slow query log

慢查询日志。该日志记录了执行时间超过阈值的 SQL 语句。该日志开启后对性能的影响不大，建议开启。

redo log

重做日志。该日志是 InooDB 引擎特有的，日志的位置在 InnoDB 存储引擎的数据目录下。

redo log 确保事务的持久性，防止在重启 MySQL 服务的时候，尚有脏页没写入磁盘。MySQL 根据 redo log 的内容进行重做，从而实现事务的持久性。

该日志记录了事务在物理层面对磁盘的修改，该日志采用追加的方式，因此写日志的操作是磁盘上一小块区域内的顺序 I/O ，而不像随机 I/O 需要在磁盘的多个地方移动磁头。事务日志持久以后，内存中被修改的数据在后台可以慢慢的刷回到磁盘。目前大多数的存储引擎都是这样实现的，我们通常称之为预写式日志。

如果数据的修改已经记录到事务日志并持久化，但数据本身还没有写回磁盘，此时系统崩溃，存储引擎在重启时能够自动恢复这部分修改的数据。具体的恢复方式则视存储引擎而定。

redo log 和 bin log 的区别

  1. bin log 是逻辑日志，记录的是可引起表数据改变的 SQL 语句的二进制形式；而 redo log 是物理日志，记录的是磁盘在物理层面对页的修改。
  2. bin log 是在事务提交完成之后一次写入，而 redo log 是在事务进行中不断写入。
  3. bin log 可作为脚本重播，整库复制；而 redo log 主要保证异常宕机或断电后的数据一致性。

🔵 MySQL 的缓冲池

概述

缓冲池就是一块内存区域，用来弥补磁盘速度较慢对性能的影响。数据库中进行读取页操作时，首先会将磁盘中读到的页放在缓冲池中，下一次再读相同的页时，如果缓冲中命中那么就直接从缓冲中读取即可。

页从缓存刷回磁盘并不是在每次页发生修改时发生，而是通过一种叫做 CheckPoint 的机制刷回磁盘。

具体来看，缓冲池中的数据包括：

  • 索引页缓存；
  • 数据页缓存；
  • redo log 缓存；
  • 插入缓存；
  • 自适应哈希索引；
  • 锁信息；
  • 数据字典信息。

其他几种缓冲都好理解，主要介绍一下 redo log 缓冲。

redo log 缓存

在下面 3 种情况下，会将 redo log 从缓存中刷回磁盘 (这里指的是将 redo log 日志本身持久化到磁盘)：

  • Master Thread 每一秒执行一次刷回
  • 每个事务提交时刷回
  • 当 redo log 缓存的剩余空间小于一半时刷回

CheckPoint 技术

如果每次一个页发生了改变，就将新页刷回磁盘，那么这个性能开销比较大。同时，如果在刷回磁盘的过程中发生了宕机，那么数据就无法恢复了。为了避免丢失的问题，MySQL 采用了先写 redo log 日志，再修改数据页的策略。由于宕机而导致数据丢失时，通过 redo log 来完成数据恢复。

当数据库发生宕机时，不需要重做所有的 redo log ，只需要对 CheckPoint 之后的日志进行恢复就行了，因为之前的页都已经刷回磁盘。

此外，当缓冲池不够用时，根据 LRU 算法会抛弃最近最少使用的页，如果这个被抛弃的页是脏的，那么就刷回磁盘，修改 CheckPoint 。

CheckPoint 无外乎是把缓存中的脏页刷回磁盘，不同之处在于每次刷回多少页，以及什么时候触发 CheckPoint 。

CheckPoint 的两种类型：

  • Sharp CheckPoint：发生在数据库关闭时将全部的脏页刷回磁盘，这是默认的。
  • Fuzzy CheckPoint：只刷回一部分脏页。

Fuzzy CheckPoint 又可以分为以下 4 种：

  • Master Thread CheckPoint ：主线程每隔一段时间刷回一部分脏页，比如每隔 1 秒或 10 秒。
  • Flush LRU List CheckPoint ：这个策略保证了 LRU 列表中需要有差不多 100 个空闲的页面，如果没有 100 个空闲的页面，那么会抛弃一些列表尾部的页面。如果被抛弃的页面是脏的，那么就要刷回磁盘。
  • Async/Sync Flush CheckPoint ：这种策略表示，在 redo log 文件不可用的情况下，需要强制将一部分页面刷回磁盘，为了保证 redo log 的循环使用性。
  • Dirty Page too much：表示脏页太多时，需要进行脏页刷回。总的来说还是为了保证缓冲池有足够多的可用的页。

🔵 MySQL 事务隔离级别和锁机制

脏读定义：读取到还未提交的数据。

不可重复读的重点在于修改和删除，幻读的重点在于添加。

隔离级别 脏读 不可重复读 幻读
Read Uncommitted 可能发生 可能发生 可能发生
Read Committed 不会发生 可能发生 可能发生
Repeatable Read 不会发生 不会发生 不会发生
Serializable 不会发生 不会发生 不会发生

共享锁 Share Lock（S 锁）

共享锁就是，对于同一数据，多个事务可以共享一把锁，都等访问到数据库，但是只能读，不能修改。

排它锁 Exclusive Lock（X 锁）

排它锁不能与别的锁共存，如果一个事务获取了一行的排它锁，其他事务就不能获得这一行的任何锁（也就不能读取或修改数据），获得了排它锁的事务才可以读取或者修改数据。

多版本并发控制（Multiversion Concurrency Control，简称 MVCC）

当一个修改数据的事务拿到了排它锁，并且修改了某一行数据，但是还没有 commit ，理论上其他的事务不可能拿到锁，也就不可能读取那一行的数据。但是，实际上其他的事务依然可以读取那一行的数据，而且读取的还是修改之前的数据。这是因为：MVCC 做了多版本的控制，另一个事务读取的实际上是一个快照。这种方式叫做一致性非锁定读取。

MVCC 使用了一些手段，让读取数据的事务，在某些时候读取的是数据库的一个快照，而写入数据（或修改数据）的事务，在 commit 之前，对于其他读数据者是不可见的！

MVCC 并发控制下的读事务一般使用时间戳或事务 ID 去标记版本。读和写相互隔离，不需要互斥。读和写并存的时候，写操作会根据目前数据库的状态，创建一个新版本，并发的读则依旧访问旧版本的数据。

MVCC 的原理细节

在 MySQL 建表的时候，每个表都有三个隐藏字段，其中两个字段与 MVCC 有关：

  1. 数据行的版本号「DB_TRX_ID」
  2. 删除版本号 「DB_ROLL_PT」

MySQL 中的每个事务都会有一个唯一的全局 ID。

  • 读取逻辑

    读取的数据行必须同时满足以下两个条件：

    1. 只选择 DB_TRX_ID 小于等于当前事务 ID 的数据行，这样可以确保事务读取的行，要么是在事务开始前已经存在的，要么是事务自身插入或者修改过的。
    2. 数据行的 DB_ROLL_PT 要么为 NULL，要么大于当前的事务 ID ，这可以确保事务能读取到在事务开始之前未被删除的行。

  • 插入逻辑

    在插入数据行的时候，会把字段 DB_TRX_ID 的值设置为执行插入操作的事务的 ID。

  • 删除逻辑

    在删除数据行的时候，并不是真正地删除，而是把 DB_ROLL_PT 字段的值置为执行删除操作的事务的 ID 。

  • 修改逻辑

    修改逻辑实际是插入逻辑和删除逻辑之和，先把删除的行的 DB_ROLL_PT 字段置为事务 ID ，然后添加一行，把 DB_TRX_ID 字段设置为事务 ID。

对锁的总结

首先，仅仅只是读取是完全不互斥的。然后，在一致性非锁定读下，即使另一个事务对某行加了 X 锁，另一个事务照常可以读取（读取快照），但是不能修改，修改操作会阻塞直到锁释放；在一致性锁定读下，如果一个事务对某行加了 X 锁，那么其他事务在读取或修改同一行时候都会阻塞。

一句话：读和读完全不互斥，读和写可以不互斥，写和写一定互斥。

行锁的 3 中算法

InnoDB 引擎有 3 种行锁的算法，分别是：

  • Record Lock：单个行记录上的锁。
  • Gap Lock：间隙锁，锁定一个范围，但是不包含记录本身。
  • Next-Key Lock：前两者之和，既锁定记录本身，也锁定一个范围。

Next-Key Lock 是前两种锁的结合，InnoDB 对于行的查询默认都是采用这种锁定算法。

然而，当查询的索引含有唯一属性的时候，Next-Key Lock 会自动降级为 Record Lock ，也就是仅仅锁住索引的那一行，而不锁住范围。若唯一索引由多个列组成，而查询仅是查找多个唯一索引列中的其中一个，那么查询其实依然是 range 类型查询，而不是 point 类型查询，故这种情况依然使用 Next-Key Lock 进行锁定。

🔵 MySQL 的索引

B+ 树

MySQL 的索引采用 B+ 树作为数据结构，所有的数据均存在叶子节点中，所有叶子节点均在同一层，而且叶子节点使用双向链表进行连接，中间节点仅仅保存指向子节点的指针。

B+ 树的优点是：分叉多，每一层的节点数量多，层数少，故 I/O 次数少。

注意，B+ 树索引并不能直接找到一个给定键值的具体行，而是只能找到被查询数据行所在的页，然后把页读如内存，再在内存中进行查找。

聚集索引和辅助索引

聚集索引的叶子节点保存行记录的所有字段；辅助索引的叶子节点只保存行记录的部分字段（但其中一定会包括主键字段）。

当查找的字段不在辅助索引中时，MySQL 会先找到对应行的主键字段，然后利用该主键字段在聚集索引中查找。这一步叫回表，会增加开销。

辅助索引又分为单列索引和组合索引。顾名思义，单列索引是建立在一个字段上的，组合索引是建立在多个字段上的。组合索引各字段中的顺序十分重要，组合索引会依次按照字段的顺序来排序数据的内容，什么意思呢？假如在 A、B、C 这三个字段上建立了组合索引（顺序为 A、B、C），那么就先按字段 A 将数据排序，A 相等的情况下，再按 B 排序，B 相等，再按 C……以此类推。比如下面几个叶子节点按照上述规则排列的：

叶子节点编号 字段 A 字段 B 字段 C
1 1 2 3
2 2 1 3
3 2 2 1
4 3 2 2
5 3 3 1
6 3 3 2

由于这种排序规律，我们可以得出结论：当 A 字段确定时，B 字段自然是有序的；A 和 B 都确定时，C 字段自然是有序的……以此类推。这个规律将作为我们书写高性能 SQL 语句的重要依据！

不应该建立索引的情况

  1. 索引列的数据区分度越高越好，所以区分度低的字段不推荐建立索引。因为索引是按照关键字值大小在 B+ 树上搜寻的，如果该字段的值大部分都相同，那么索引就会退化为线性查找，失去优势。比如自增主键 id 的区分度是最高的（因为每一行的 id 都不一样），所以在 MySQL 中尽量所有的 update 都使用主键 id 去更新。
  2. 频繁更新的字段不建议建立索引。因为更新值的同时也要更新 B+ 树，这样会加大开销。
  3. where 条件用不到的字段不需要建立索引。

应该建立索引的情况

  1. where 子句经常用到的字段应建立索引。
  2. order by 子句经常用到的字段应建立索引。
  3. 关联查询时，外键字段应建立索引。

🔵 explain 语句

explain 关键字用来分析 SQL 语句，其输出结果的各个字段解释如下：

id

表示 SQL 语句的执行顺序。

select_type

表示 SQL 查询的类型，若是简单查询则值为 Simple。其他可能的值有 Primary、Union、Derived……

table

当前表名。

partitions

匹配的分区。

type

性能指标，这个字段最重要！性能从好到差依次如下：

  1. system：表中只有一行数据。
  2. const：单表中最多有一个匹配行，即对 primary key 或 unique index 的定值检索。
  3. eq_ref：多表连接中被驱动表的连接列上有 primary key 或者 unique index的检索。
  4. ref：常见！在索引列上的定值查询。
  5. range：范围查询，表示使用索引从表中查询出一些数据。
  6. index：表示在辅助索引树上的全树扫描，比全表扫描略好。
  7. all：全表扫描。

possible_keys

提示哪个索引可能会被用来查询所需的数据。

key

实际使用到的索引。

key_len

用到索引的长度。

ref

表示使用哪个定值从表里寻找行。（若 where 子句条件是范围，则该字段为 null）

rows

表示得出最终查询结果需要扫描的记录数。这个值越小越好。

filtered

过滤的百分比。

extra

一些额外的说明，几个常见的值如下；

  • Using filesort：利用 filesort 进行 order by 排序。
  • Using index：使用到辅助索引。
  • Using where：在存储引擎检索出记录后，在 server 利用 where 条件进行过滤，并返回给客户端。

🔵 MySQL 的分区

概述

就访问数据库的应用而言，从逻辑上讲，只有一个表或一个索引，但是在物理上这个表或索引可能由很多个物理文件分区组成，每个分区都是独立的对象，可以独自处理，也可以作为一个更大的对象处理。

MySQL 的分区只支持拆分行，并不支持拆分列，此外，MySQL 的每个分区中既保存了数据，也保存了索引，并不是数据保存在分区，而索引统一管理。

MySQL 在进行分区时，需要指定一个分区字段作为标准，然后根据该字段上不同的值以及事先定好的策略把行记录放入不同的分区中。注意：不论以何种方式建立的分区，分区列必须是唯一索引的一个组成部分，否则就无法分区。

MySQL 的分区类型有：

  • Range 分区：行数据基于一个给定的连续区间的列值被放入分区。
  • List 分区：和 Range 分区相似，只不过 List 分区面向的是离散的值。
  • Hash 分区：根据自定义的表达式的返回值来进行分区，返回值不能为负数。
  • Key 分区：根据 MySQL 数据库提供的哈希函数来分区。

分区的好处：

  • 可以在一张表中存储大于单个物理磁盘容量的数据，因为我们可以把一张表的不同分区存放在不同物理磁盘上。
  • 对于已经过期或不需要的数据，可以通过直接删除分区来达到快速删除数据的目的。
  • 优化查询，在 where 子句中包含分区条件时，可以只扫描必要的一个或多个分区来提高查找效率。
  • 涉及到聚集函数 sum()、count() 的查询时，可以在每个分区上并行处理。
  • 凭借在多个磁盘上传播数据，可以实现更高的查询吞吐量。

Range 分区

比如指定分区列为 id 字段，当 id < 10 时，放入分区一；当 id > 10 且 <20 时，放入分区二。如果插入的值不在分区列的定义中，那么会抛出异常。这种方式非常适合按照日期列来分区，比如对于销售表，可以根据年份来存放销售记录，在对于某个年份的记录进行操作时，就会变得很方便。

List 分区

这种方式和 Range 分区相似，只不过是离散的，不是连续的。对于指定的分区列的值，放入指定的分区，同样，如果插入的值不在分区列的定义中，那么会抛出异常。

Hash 分区

这种方式的目的是把行记录均匀地分布在每个分区中，需要自定义哈希函数。如果对于值连续的列 (比如自增长主键) 进行 Hash 分区，可以更好地将数据进行平均分布。

Key 分区

和 Hash 分区相似，只不过不用自定义哈希函数，而是使用 MySQL 数据库提供的函数进行分区。

🔵 慢 SQL 的优化

检查 select 后的字段

  1. select 后使用了用户自定义函数时，SQL 返回多少行，该函数就会被调用多少次，非常影响性能。
  2. select 后出现了 text 类型的字段，会消耗大量的 IO 带宽，甚至该字段内容过大，超过 max_allowed_packet 时，会出现报错。
  3. 在 select 中含有另一个子查询的情况叫做内联子查询，SQL 返回多少行，子查询就要执行多少次，非常影响性能。

检查 where 子句

  1. where 子句中的索引字段不要参与计算，否则会导致无法走索引。
  2. 注意隐式的类型转化，比如 int 型转换为 varchar 型，这也会导致无法走索引。

检查 order by 子句

  1. order by 后面使用索引字段，可以充分利用索引本身的有序性来提升性能（前缀索引除外）。
  2. order by 后面的字段应该按照组合索引的顺序来排列。

检查索引

  1. 恰当使用前缀索引。比如 varchar 类型如果太长，可以考虑创建前缀索引来减少索引列长度，但是前缀索引无法消除 order by 带来的排序开销。
  2. 对于域名这一类的字段，由于前缀重复度高，而且整体很长，我们可以把域名先倒序（制造区分度），然后建立前缀索引（减少长度）。

🔵 MySQL 常见问题

主键用自增还是 UUID ？

  • 如果使用自增，那么每次插入一条新的记录，记录就会顺序添加到最后一条记录 (叶子节点) 的后续位置，如果一页写满，那么就新开辟一页。
  • 如果使用 UUID，由于每次插入新数据的主键值随机，所以每次都要插入到聚集索引树的一个随机位置，可能会加大 I/O 开销。

主键为什么不推荐有业务含义？

因为任何有业务含义的列都有改变的可能性，主键一旦带上了业务含义，那么主键就有可能发生变更。主键一旦发生变更，该数据在磁盘上的存储位置就会发生变更，有可能会引发页分裂，产生空间碎片。

还有就是，带有业务含义的主键，不一定是顺序自增的。那么就会导致数据的插入顺序，并不能保证后面插入数据的主键一定比前面的数据大。如果出现了，后面插入数据的主键比前面的小，就有可能引发页分裂，产生空间碎片。

货币字段用什么类型？

用 Decimal 类型，因为 float 和 double 都会有误差。

where 的执行顺序是怎样的？

where 条件从左往右执行的，在数据量小的时候不用考虑，但数据量多的时候要考虑条件的先后顺序，此时应遵守一个原则：能排除越多数据行的条件放在越前面。
