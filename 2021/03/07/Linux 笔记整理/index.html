🔵前序知识：磁盘分区

磁盘连接的方式与设备文件名的关系

现在个人计算机常见的磁盘接口有两种，分别是 SATA 与 SAS 接口，主流是 SATA 接口。以 SATA 接口来说，由于 SATA/USB/SAS 等磁盘接口都是使用 SCSI 模块来驱动的， 因此这些接口的磁盘设备文件名都是 /dev/sd[ap] 的格式。

MBR (Master Boot Record) 与 GPT (GUID Partition Table) 磁盘分区

  • MBR：

    早期的 Linux 系统为了相容于 Windows 的磁盘，因此使用的是支持 Windows 的 MBR 的方式来处理开机管理程序与分区表。开机管理程序纪录区与分区表则通通放在磁盘的第一个扇区， 这个扇区有这两部分数据：

    • 主要开机记录区 (Master Boot Record) ：可以安装开机管理程序的地方。

    • 分区表 (partition table) ：记录整颗硬盘分区的状态。

      MBR 分区有以下特点：

    • 所谓的“分区”只是针对分区表部分进行设置而已。

    • 分区表所能记录的分区有主要分区 (primary) 和延伸分区 (extended) 。

    • 延伸分区最多有一个。

    • 其中由于分区表的大小有限，所以能够记录的分区信息也有限，然而已经分出来的延伸分区可以再分为若干个逻辑分区。

  • GPT：

    为了相容于所有的磁盘，因此在扇区的定义上面，GPT 使用逻辑区块位址 (Logical Block Address, LBA) 来规划，第一个 LBA 称为 LBA0。

    GPT 使用了 34 个 LBA 区块来记录分区信息。与 MBR 仅用一个区块来记录分区信息，损坏就死光光的情况不同，GPT 除了前面 34 个 LBA 之外，整个磁盘的最后 33 个 LBA 也拿来作为另一个备份，这样大大增强安全性。

    上图的解释如下：

    • LBA0 (MBR 相容区块)：与 MBR 模式相似的，这个相容区块也分为两个部份，相同的是，一部分存放开机管理程序；不同的是，在原本应该存放分区记录表的地方，仅存放一个特殊标志的分区，用来标识此磁盘为 GPT 格式，而不认识 GPT 分区表的磁盘管理程序， 就不会认识这颗磁盘，除非用户有特别要求要处理这颗磁盘，否则该管理软件不能修改此分区信息，进一步保护了磁盘。

    • LBA1 (GPT 表头纪录)：这个部份记录了分区表本身的位置与大小，同时记录了备份用的 GPT 分区 (就是后面的 32 个 LBA 区块) 放置的位置， 同时放置了分区表的检验机制码，操作系统可以根据这个检验码来判断 GPT 是否正确。若有错误，还可以通过这个纪录区来取得备份的 GPT 来恢复 GPT 的正常运行。

    • LBA2-33 (实际纪录分区信息处)：每个 LBA 都可以记录若干个分区信息。

      GPT 分区已经没有所谓的主、延伸、逻辑分区的概念，既然每笔记录都可以独立存在， 当然每个都可以视为是主分区，每一个分区都可以格式化。

开机流程中的 BIOS 与 UEFI 开机检测程序

没有执行软件的硬件是没有用的，所以就有了操作系统，用来管理所有的硬件，使计算机能够认识硬盘内的文件系统，并且进一步的读取硬盘内的软件文件与执行该软件来达成各项软件的执行目的。

问题是，操作系统本身也是软件，那么我的计算机又是如何认识这个操作系统软件并且执行他的？明明开机时我的计算机还没有任何软件系统，那他要如何读取硬盘内的操作系统文件？这就得要牵涉到计算机的开机程序了，主要有 BIOS 和 UEFI 两种机制。

  • BIOS 搭配 MBR/GPT 的开机流程：

    BIOS 是一个写入到主板上的一个固件 (固件就是写入到硬件上的一个软件程序)。这个 BIOS 就是在开机的时候，计算机系统会主动执行的第一个程序。

    对于 MBR 磁盘，整个开机流程大致是以下过程：

    1. 开机自动执行 BIOS 固件，认识第一个可开机的设备；

    2. 运行 MBR 第一个扇区中内含的开机管理程序；

    3. 开机管理程序读取核心文件；

    4. 操作系统开始运行并且接管后面所有的事情。

      对于 GPT 磁盘，整个过程类似，只不过需要 BIOS 能够识别出 GPT 即可。

      开机管理程序 (boot loader) 主要有下面一些功能：

    • 提供菜单：使用者可以选择不同的开机项目，这也是多系统开机的重要功能。

    • 载入核心文件：直接指向可开机的程序区段来开始操作系统。

    • 转交其他 loader：将开机管理功能转交给其他 loader 负责。

      第三点表示计算机系统里面可能具有两个以上的开机管理程序。开机管理程序并不是只能存放在 MBR 的第一个扇区中，每一个分区里都能存放开机管理程序，比如在一个分区中放置 Windows 的开机管理程序，在另一个分区中放置 Linux 的开机管理程序。

  • UEFI 搭配 GPT 的开机流程：

    UEFI 使用 C 程序语言，比起使用汇编语言的传统 BIOS 要更容易开发，也因为使用 C 语言来撰写，因此如果开发者够厉害，甚至可以在 UEFI 开机阶段就让该系统了解 TCP/IP 而直接上网，根本不需要进入操作系统，这让小型系统的开发充满各种可能性。

比较项目 传统BIOS UEFI
使用的语言 汇编语言 C 语言
硬件资源控制 使用中断（IRQ） 管理不可变的内存存取不可变得输入/输出存取 使用驱动程序与协定
处理器运行环境 16 位 CPU 保护模式
扩充方式 通过 IRQ 链接 直接载入驱动程序
第三方厂商支持 较差 较好且可支持多平台
图形化能力 较差 较好
内置简化操作系统前环境 不支持 支持

🔵Linux 简介

Linux 有 3 种不同的接口：

  1. 真正的系统调用接口
  2. 库函数接口
  3. 由标准应用程序 (如 shell) 构成的接口

Linux 的内核结构：

  • IO 部件：从顶层看，这些 IO 功能全整合在一个虚拟文件系统层中，以顶层的视角，对一个文件进行读操作，不论是在内存中还是在磁盘中，都和从终端输入中读取一个字符是一样的；从底层看，所有的 IO 操作最终都要通过某个设备驱动器。
  • 内存管理部件：功能包括维护虚拟内存到物理内存的映射，实现一个好的页面置换算法等。
  • 进程管理部件：主要的任务是进程的创建和终止，还有进程的调度。

处在 3 个部件最顶层的是内核的系统调用接口，所有的系统调用都来自于这里，其触发一个陷入，并将系统从用户态转换到受保护的内核态，从而将控制权交给上述的部件之一。

🔵Linux 中的进程

简述

Linux 中创建新进程的方法是 fork 系统调用。调用 fork 函数的进程叫父进程，新的进程叫做子进程。fork 函数给父进程返回一个非零值作为子进程的 PID，而给子进程返回 0。也就是父进程会知道子进程的 PID，如果子进程想知道自己的 PID，可以调用 getpid。

fork 刚结束调用时，父、子进程的内存映像、变量、寄存器以及其他所有东西都是相同的，然后它们各循其路。

在 Linux 系统内核中，进程通过数据结构 task_struct 被表示成任务 (task)，task_struct 存在于内存中。不像其他的操作系统会区别进程、轻量级进程和线程，Linux 系统用任务的数据结构来表示所有的执行上下文。所以，一个单线程的进程只有一个任务数据结构，而一个多线程的进程会为每一个用户级线程分配一个任务数据结构。最后，Linux 内核是多线程的，并且它所拥有的是与任何用户线程无关的内核级线程，这些内核线程执行内核代码。

进程组

一个 session 可能会有一个 session 首进程，而一个 session 首进程可能会有一个控制终端。

一个进程组可能会有一个进程组首进程。进程组首进程的 PID 与该进程组的 PID 相同。

与终端交互的进程是前台进程，否则便是后台进程。

进程间的信号

一个进程可以给另一个进程发送信号，用来告诉操作系统当信号到来时它们希望发生什么事件。一个进程只可以给它所在的进程组中的进程发送信号。

常见的信号有 SIGHUP（挂断），接受到该信号的进程会终止自己。

SIGHUP 会在下面 3 种情况下发送给相应的进程：

  1. 终端关闭时，该信号被发送到 session 首进程以及作为 job 提交的进程（也就是用 & 符号提交的进程）
  2. session 首进程退出时，该信号被发送到该 session 中的前台进程组和后台进程组中的每一个进程
  3. 若进程的退出，导致一个进程组变成了孤儿进程组，且新出现的孤儿进程组中有进程处于停止状态，则 SIGHUP 和 SIGCONT 信号会按顺序先后发送到新孤儿进程组中的每一个进程。

& 符号：

& 符号就是将程序放在后台运行，也就是作为一个 job 来运行。如果此时（该命令还没完全执行结束）终端断开，程序还是会被 SIGHUP 信号终止，也就是上面提到的第 1 种场景。

但是如果此时在终端中执行 exit 命令，就会使得当前主进程退出。从而使得后台执行的程序成为孤儿进程，从而被 init 进程接管。此时终端再断开就 不会受到 SIGHUP 信号了。这时即使终端断开，程序也会继续运行。这也是为什么我们在一个 shell 脚本中通过 & 执行一个程序后，shell 脚本执行完，即使终端断开，shell 脚本启动的后台程序也不会退出。因为 shell 执行完后，其启动的程序就已经成为了孤儿进程，从而被 init 接管了。

nohup 命令的原理：

nohup 的格式为 nohup command &

由 nohup 命令执行的 command 程序会忽略所有的 SIGHUP 信号，从而不会在终端关闭或退出时由于 SIGHUP 信号导致程序退出。

nohup 本身不会将程序放入后台运行，但是通过 nohup 方式启动的程序，由于其忽略了 SIGHUP 信号，其接收到 SIGHUP 信号也不会退出。

🔵 命令行

两个常用按键

[Ctrl]-c 按键：当某个程序在中断窗口跑不停时，可以使用这个按键来强行停止。

[Ctrl]-d 按键：这个按键可以终止键盘输入，类似 exit。

使用 “–help” 求助

基本上，如果是指令，那么通过这个简单的 –help 就可以很快速的取得你所需要的选项、参数的说明，不过，通常 –help 用在协助你查询“你曾经用过的指令所具备的选项与参数”而已， 如果你要使用的是从来没有用过得指令，或者是你要查询的根本就不是指令，而是文件的“格式”时，那就得要通过 man page 。

使用 man page 求助

man + 指令的输出结果中，在指令后面会有一个数字，这个数字代表的意义如下：

代号 含义
1 使用者在 shell 环境中可以操作的指令或可执行文件
2 系统核心可调用的函数与工具等
3 一些常用的函数 (function) 与函数库 (library) ，大部分为 C 的函数库 (libc)
4 设备文件的说明，通常在 /dev 下的文件
5 配置文件或者是某些文件的格式
6 游戏 (games)
7 惯例与协定等，例如 Linux 文件系统、网络协定、ASCII code 等等的说明
8 系统管理员可用的管理指令
9 跟 kernel 有关的文件

输出结果中，也分成了好几个部分，大致如下：

代号 内容说明
NAME 简短的说明
SYNOPSIS 简短的指令下达语法 (syntax) 简介
DESCRIPTION 较为完整的说明，这部分最好仔细看看
OPTIONS 针对 SYNOPSIS 部分中，有列举的所有可用的选项说明
COMMANDS 当这个程序（软件）在执行的时候，可以在此程序（软件）中下达的指令
FILES 这个程序或数据所使用或参考或链接到的某些文件
SEE ALSO 可以参考的，跟这个指令或数据有相关的其他说明
EXAMPLE 一些可以参考的范例

对于输出结果的查阅技巧：

  1. 先察看 NAME 部分，约略看一下这个指令的意思
  2. 再详看一下 DESCRIPTION，这个部分会提到很多相关的数据与使用时机，从这个地方可以学到很多小细节
  3. 而如果这个指令本身就很熟悉，那么主要就是查看 OPTIONS 的部分了，可以知道每个选项的意义，这样就可以下达比较细部的指令内容

🔵 正确的关机方法

由于装有 Linux 操作系统的服务器一般是多人使用的，所以关机时必须注意以下问题：

  • 用户在线状态：使用 who 命令观察有哪些用户还在线。
  • 网络连接状态：使用 netstat -a 命令观察网络连接。
  • 进程状态：使用 ps -aux 观察进程状态。

🔵 权限

权限对于一般文件的含义

文件是实际含有数据的地方，包括一般文本文件、数据库内容档、二进制可可执行文件等等。因此，权限对于文件来说，他的意义是这样的：

  • r ：可读取此文件的实际内容，如读取文本文件的文字内容等。
  • w ：可以编辑、新增或者是修改该文件的内容（但不含删除该文件）。
  • x ：该文件具有可以被系统执行的权限。

权限对于目录的含义

  • r ：表示具有读取目录中文件清单的权限，也就是可以利用 ls 这个指令将该目录的内容列表显示出来。
  • w ：表示具有异动该目录结构清单的权限，包括：在其中创建新的文件或目录、删除已经存在的文件或目录、将已经存在的文件或目录重命名、搬移该目录中的文件或目录的位置。
  • x ：表示能否进入目录的权限，也就是能否使用 cd 命令切换进目录，把目录当成抽屉的话，这就相当于抽屉的钥匙。

总结

对于一般文件，权限主要是针对“某具体文件中的内容”；对于目录，权限针对的是“目录内的文件”。

权限与指令间的关系

使用者进入某目录需要的基本权限包括：

  • 使用者对这个目录至少需要具有 x 的权限。
  • 如果使用者想要在这个目录内利用 ls 列出文件列表，则使用者对此目录还需要 r 的权限。

使用者在某个目录内读取一个文件需要的基本权限包括：

  • 使用者对这个目录至少需要具有 x 权限。
  • 使用者对该文件至少需要具有 r 的权限。

使用者修改一个文件需要的基本权限包括：

  • 使用者在该文件所在的目录至少要有 x 权限。
  • 使用者对该文件至少要有 r, w 权限。

使用者创建一个文件需要的基本权限包括：

  • 使用者在该目录要具有 w,x 的权限，重点在 w。

🔵 目录规定以及 FHS 标准

特定的文件必须有规律地放入特定的目录中。FHS (Filesystem Hierarchy Standard) 标准指出，目录有四种类型：

  • 可分享的：可以分享给其他系统挂载使用的目录，所以包括可执行文件与使用者的邮件等数据， 是能够分享给网络上其他主机挂载用的目录。
  • 不可分享的：自己机器上面运行的设备文件或者是与程序有关的 socket 文件等， 由于仅与自身机器有关，所以当然就不适合分享给其他主机了。
  • 不变的：有些数据是不会经常变动的，不随着发行版而变动。例如函数库、文件说明文档、系统管理员所管理的主机服务配置文件等等。
  • 可变动的：经常改变的数据，例如登录文件、一般用户可自行收受的新闻群组等。

必须存在的目录：

目录 应放置的文件内容
/bin 系统有很多放置可执行文件的目录，但 /bin 比较特殊。因为 /bin 放置的是在单人维护模式下还能够被操作的指令。在 /bin 下面的指令可以被 root 与一般帐号使用，主要有：cat, chmod, chown, date, mv, mkdir, cp, bash 等等常用的指令。
/boot 这个目录主要在放置开机会使用到的文件，包括 Linux 核心文件以及开机菜单与开机所需配置文件等等。Linux kernel 常用的文件名为：vmlinuz，如果使用的是 grub2 这个开机管理程序， 则还会存在 /boot/grub2 这个目录喔！
/dev 在 Linux 系统上，任何设备与周边设备都是以文件形式存在于这个目录当中的。你只要通过存取这个目录下面的某个文件，就等于存取某个设备。比要重要的文件有 /dev/null，/dev/zero， /dev/tty，/dev/loop， /dev /sd 等等。
/etc 系统主要的配置文件几乎都放置在这个目录内，例如人员的帐号密码档、 各种服务的启始档等等。一般来说，这个目录下的各文件属性是可以让一般使用者查阅的， 但是只有 root 有权力修改。FHS 建议不要放置可执行文件 (binary) 在这个目录中。比较重要的文件有： /etc/modprobe.d/，/etc/passwd，/etc/fstab，/etc/issue 等等。另外 FHS 还规范几个重要的目录最好要存在 /etc/ 目录下：/etc/opt（必须存在）：这个目录中放置第三方软件的相关配置文件。
/lib 系统的函数库非常的多，而 /lib 放置的则是在开机时会用到的函数库， 以及在 /bin 或 /sbin 下面的指令会调用的函数库而已。什么是函数库呢？你可以将他想成是“外挂”，某些指令必须要有这些“外挂”才能够顺利完成程序的执行之意。另外 FSH 还要求下面的目录必须要存在：/lib/modules/，这个目录主要放置可抽换式的核心相关模块（驱动程序）。
/media media 是“媒体”的英文，顾名思义，这个 /media 下面放置的就是可移除的设备，包括软盘、光盘、DVD 等等设备都暂时挂载于此。常见的文件名有：/media/floppy， /media/cdrom 等等。
/mnt 如果你想要暂时挂载某些额外的设备，一般建议你可以放置到这个目录中。在古早时候，这个目录的用途与 /media 相同啦！只是有了 /media 之后，这个目录就用来暂时挂载用了。
/opt 这个是给第三方软件放置的目录。什么是第三方软件啊？举例来说，KDE 这个桌面管理系统是一个独立的计划，不过他可以安装到 Linux 系统中，因此 KDE 的软件就建议放置到此目录下了。另外，如果你想要自行安装额外的软件（非原本的 distribution 提供的），那么也能够将你的软件安装到这里来。不过，以前的 Linux 系统中，我们还是习惯放置在 /usr/local 目录下呢！
/run 早期的 FHS 规定系统开机后所产生的各项信息应该要放置到 /var/run 目录下，新版的 FHS 则规范到 /run 下面。由于 /run 可以使用内存来仿真，因此性能上会好很多！
/sbin Linux 有非常多指令是用来设置系统环境的，这些指令只有 root 才能够利用来“设置”系统，其他使用者最多只能用来“查询”而已。放在 /sbin 下面的为开机过程中所需要的，里面包括了开机、修复、还原系统所需要的指令。至于某些服务器软件程序，一般则放置到 usr/sbin/ 当中。至于本机自行安装的软件所产生的系统可执行文件 (system binary)， 则放置到 /usr/local/sbin/ 当中了。常见的指令包括：fdisk，fsck，ifconfig，mkfs 等等。
/srv srv 可以视为“service”的缩写，是一些网络服务启动之后，这些服务所需要取用的数据目录。常见的服务例如 WWW, FTP 等等。举例来说，WWW 服务器需要的网页数据就可以放置在 /srv/www/ 里面。不过，系统的服务数据如果尚未要提供给网际网络任何人浏览的话，默认还是建议放置到 /var/lib 下面即可。
/tmp 这是让一般使用者或者是正在执行的程序暂时放置文件的地方。这个目录是任何人都能够存取的，所以你需要定期的清理一下。当然，重要数据不可放置在此目录啊！因为 FHS 甚至建议在开机时，应该要将 /tmp下的数据都删除。

/var 第二层 FHS 设置，主要为放置变动性的数据，后续介绍

建议存在的目录：

目录 应该放置的内容
/home 这是系统默认的使用者主文件夹 (home directory)。在你新增一个一般使用者帐号时， 默认的使用者主文件夹都会规范到这里来。比较重要的是，主文件夹有两种代号：~ 代表目前这个使用者的主文件夹；~user ：则代表 user 的主文件夹！
/lib 用来存放与 /lib 不同的格式的二进制函数库，例如支持 64 位的 /lib64 函数库等
/root root 用户的文件夹。之所以放在这里，是因为如果进入单人维护模式而仅挂载根目录时， 该目录就能够拥有 root 的主文件夹，所以我们会希望 root 的主文件夹与根目录放置在同一个分区中。

有两个目录需要特别注意：

  • /usr ：这个目录其实是 Unix Software Resource 的缩写，表示 Unix 系统软件资源。FHS 建议所有的软件开发者应该将他们的数据合理放置到这个目录下的次目录。
  • /var ：如果说 /usr 是安装时会占用较大硬盘容量的目录，那么 /var 就是在系统运行后逐渐占用硬盘容量的目录。该目录主要针对常态性变动的文件，包括 cache、log file 以及某些软件运行所产生的文件，例如 MySQL 数据库文件等等。

查看文件内容

概述：

  • cat ：显示整个文件的内容。
  • tac ：从最后一行开始显示整个文件的内容。
  • more ：分页显示文件内容。
  • less ：与 more 类似，不同的是，less 可以往前翻页。
  • head ：只查看文件的头几行。
  • tail ：只查看文件的末几行。
  • od ：以二进制的方式读取文件内容。

more 命令：

  • space 键：向下翻一页。
  • Enter 键：向下翻一行。
  • / + 字符串：搜寻一个字符串。
  • q 键：停止显示文件内容。

less 命令：

  • space 键：向下翻一页。
  • pagedown 键：向下翻一页。
  • pageup 键：向上翻一页。
  • / + 字符串：向下搜索字符串。
  • ? + 字符串：向上搜索字符串。
  • q 键：停止显示文件内容。

搜索文件

find 命令一般不常用，因为它是直接搜寻硬盘。一般使用 whereis 和 locate 命令来定位文件，如果无法找到，才会使用 find 命令来查找。因为 whereis 只会找系统中某些特定目录下面的文件，而 locate 使用数据库来搜寻文件。

whereis：

whereis 只会找系统中某些特定目录下面的文件，主要是针对 /bin 和 /sbin 下面的可执行文件， 以及 /usr/share/man 下面的 man page 文件。

locate：

使用数据库来搜寻，已创建的数据库放在 /var/lib/mlocate/ 中。此数据库的创建是每天一次（每个发行版略有不同），更新数据库使用 updatedb 命令。

find：

直接在磁盘中遍历搜索，会很慢。

🔵 文件系统

Linux 的 Ext 文件系统是索引式文件系统，使用 i-node 来记录所用到的磁盘块号，总共有三部分：

  • superblock ：记录此文件系统的整体信息，包括 inode/block 的总量、使用量、剩余量，以及文件系统的格式与相关信息等。
  • i-node ：一个文件占用一个 i-node ，存放文件的属性和用到的 block 号。
  • block ：存放文件的实际内容，若文件太大，会占用多个 block。

对于每个文件（不管是目录还是普通文件），都会占用一个 i-node。而对于目录来说，由于其只存放文件名，一般初始只会得到一个 block，目录的 i-node 记录了这个 block 的号码，然后 block 中存放的是目录中文件的 i-node 号码。如果某目录下的文件太多，该目录可能也会占用不止一个 block。

🔵 Linux 中的 I/O 模式

关于 I/O 的基本概念

  • 同步与异步：当一个同步调用发出后，调用者一直要等待返回结果通知后，才能进行后续的执行；当一个异步调用发出后，调用者不能立刻得到返回的结果通知，实际处理这个调用的部件在完成后，通过状态、通知或回调来告知调用者可以继续执行。
  • 阻塞和非阻塞：这两个概念主要是从线程等待消息通知时的状态角度来说的，无所谓同步异步。阻塞指得到调用结果之前，线程会被挂起；非阻塞则线程不会被挂起，立即返回。

总结：同步就是指啥事都调用者负责；异步就是把事情分一部分给被调用者。阻塞就是调用者线程干等，其他啥也不干；非阻塞就是调用者线程可以干其他事情，然后时不时看一眼调用是否产生结果并结束。同步与异步关注的是消息通知机制，阻塞非阻塞关注的是等待消息通知时，线程的状态。

文件描述符与 I/O 模式

文件描述符 (File Description) 是计算机中的一个术语，是一个指向已打开文件的引用。fd 在形式上是一个非负整数，实际上它是一个索引值，指向内核中为每一个进程所维护的该进程已打开文件的记录表。当进程打开一个现有文件或者创建一个新文件时，内核向该进程返回一个文件描述符。在涉及底层的程序编写时，常常会围绕 fd 展开。注意：fd 这一概念只适用于类 UNIX 系统，不适用于 Windows 系统。

在 Linux 中，I/O 全过程有 2 个阶段：

  1. 等待数据到达并进入内核缓冲区。
  2. 将数据从缓冲区搬迁到用户进程地址空间。

正因为这两个阶段，Linux 系统主要产生了 4 种网络模式的方案：

  • 阻塞 I/O
  • 非阻塞 I/O
  • I/O 多路复用
  • 异步 I/O

阻塞 I/O ：默认情况下的阻塞 I/O 都是阻塞的，例如 recvfrom 系统调用，直到整个阻塞 I/O 过程完成之前，也就是数据复制到进程地址空间之前，该进程都是处于阻塞状态。

**非阻塞 I/O (NOBLOCK)**：进程采用轮询的机制，当等待数据到达时，进程不会阻塞，会一直轮询，但是数据到达后，从内核空间拷贝到进程空间，这一阶段会阻塞。

**I/O 多路复用(multiplexing)**：又叫做事件驱动 I/O，主要是 select 、poll 系统调用实现的。阻塞发生在 select/poll 操作上， select/poll 不断轮询多个 fd (网络情况下就是多个 socket)，当某个 socket 有数据到达并准备好了，那么就通知进程再调用 recvfrom 将数据复制到进程空间。支持的 fd 数量有限。

异步 I/O (AIO) ：使用 aio_read 调用，给内核传递描述符、缓冲区指针、缓冲区大小和文件偏移量，并告诉内核，当整个 I/O 过程完成后通知进程。该系统调用立即返回，期间进程完全不会被阻塞。

I/O 多路复用有关的系统调用

select ：

select 可以同时监视多个 fd，调用 select 的进程会被阻塞，直到有文件描述符就绪、或者超时，此时 select 函数返回，然后可以通过遍历 fd，来找到就绪的描述符。

select 的缺点有：

  • 单个进程可以监听的 fd 数量有上限，一般是 1024 个。
  • 对 socket 的扫描采用的是轮询的方式，效率比较低。
  • 需要维护一个用来存放大量 fd 的数据结构，这样会使得用户空间和内核空间在传递该结构时开销较大。

poll：

poll 与 select 类似，有些许差别。

  • 大量的 fd 数组被整体复制于用户态和内核态地址空间之间，而不管这样的复制是不是有意义。
  • 有“水平触发”的特点，如果报告了 fd 后，fd 没有被处理，那么下次会继续报告该 fd。

epoll：

  • 能够监听的 fd 没有上限。
  • 效率提升，不是采用轮询的方式，当 fd 数量增加时，效率并不会下降。
  • 从用户空间拷贝到内核空间时，运用了 mmap() 映射减少了复制开销。

支持高并发的网络 I/O 编程技术

在高 TCP 并发的情形下，如果使用同步 I/O，这会严重阻塞程序的运转，除非为每个 TCP 连接的 I/O 创建一个线程。但是，过多的线程又会因系统对线程的调度造成巨大开销。

因此，在高 TCP 并发的情形下使用同步 I/O 是不可取的，这时可以考虑使用 I/O 多路复用或异步 I/O。

🔵 守护进程 (daemon) 和服务 (service)

概述

简单的说，系统为了某些功能必须要提供一些常驻后台的服务（不论是系统本身还是网络方面），这个服务就称为 service，达成这个 service 的程序我们就称呼他为 daemon，事实上，这两者不用作很严格的区分。

每一个 daemon 的开发者，当初在开发他们的服务时，名称后通常都会加上一个 d，这个 d 就是代表 daemon。所以，我们使用了 ps 与 top 来观察程序时，都会发现到很多的 {xxx}d 的程序，通常那就是一些daemon 程序。

系统核心第一支调用的程序是 init ，然后 init 去唤起所有的系统所需要的服务。

  • 启动脚本复杂：init 进程只是执行启动脚本，不管其他事情。脚本需要自己处理各种情况，这往往使得脚本变得很长。

systemd 对 daemon 的管理

systemd 的好处：

  • 平行处理所有服务，加速开机流程。
  • 一经要求就回应的 on-demand 启动方式： systemd 使用主命令 systemctl 来处理，基本无须其他额外的指令来支持。
  • 服务依赖性的自动检查： 由于 systemd 可以自动进行服务依赖性的检查，因此如果 B 服务必须在在 A 服务启动后启动，那当你在没有启动 A 服务的情况下仅手动启动 B 服务时， systemd 会自动帮你启动 A 服务。
  • 按照 daemon 功能分类：systemd 管理的服务非常多，首先 systemd 将所有的服务定义为**服务单元 (Unit)**，并将各个 Unit 归类到不同的类型 (type) 中去。旧的 init 仅分为 stand alone 与 super daemon，实在不够看，systemd 将 Unit 区分为 service，socket，target，path，snapshot，timer 等多种不同的类型， 方便管理员的分类与记忆。

Unit 的分类（按照文件扩展名）：

  • service：一般服务类型：主要是系统服务，包括服务器本身所需要的本机服务以及网络服务都是，比较经常被使用到的服务大多是这种类型，所以，这也是最常见的类型了。
  • .target：一组 Unit 的集合，也就是说， 选择执行 XXX.target 就是执行一堆其他 .service 之类的服务。
  • .socket：内部程序数据交换的套接字服务：主要是 IPC (Inter-process communication) 的传输讯息套接字功能。一般用于本机服务比较多，例如我们的图形界面很多的软件都是通过 socket 来进行本机程序数据交换的行为。
  • .mount .automount：文件系统挂载相关的服务 (automount unit / mount unit)：例如来自网络的自动挂载、NFS 文件系统挂载等与文件系统相关性较高的程序管理。
  • .path：侦测特定文件或目录类型 (path unit)：某些服务需要侦测某些特定的目录来提供服务，例如最常见的打印服务，就是通过侦测打印伫列目录来启动打印功能，这时就得要 .path 的服务类型支持了。
  • .timer：循环执行的服务 (timer unit)。

Unit 的操作：

每个 Unit 都有一个配置文件，告诉 systemd 怎么启动这个 Unit，默认从 /etc/systemd/system/ 这个目录下读取配置文件，但是里面存放的大部分文件都是符号链接，指向其他地方的真正的配置文件。

操作 命令
列出正在运行的 Unit systemctl list-units
列出所有 Unit，包括没有找到配置文件的或者启动失败的 systemctl list-units –all
列出所有正在运行的、类型为 service 的 Unit systemctl list-units –type=service
显示单个 Unit 的状态 systemctl status XXX.service
启动一个服务 sudo systemctl start XXX.service
停止一个服务 sudo systemctl stop XXX.service
重启一个服务 sudo systemctl restart XXX.service
杀死一个服务的所有子进程 sudo systemctl kill XXX.service
重新加载一个服务的配置文件 sudo systemctl reload XXX.service
重载所有修改过的配置文件 sudo systemctl daemon-reload
在两个目录之间建立链接 sudo systemctl enable XXX.service
在两个目录之间撤销链接 sudo systemctl disable XXX.service
列出所有配置文件及其状态 systemctl list-unit-files

Unit 的 4 个状态：

  • enabled：已建立启动链接。
  • disabled：没建立启动链接。
  • static：该配置文件没有 [Install] 部分（无法执行），只能作为其他配置文件的依赖。
  • masked：该配置文件被禁止建立启动链接。

Unit 配置文件的内容解释：

配置文件的内容一般分为几个区块：

  1. [Unit] 区块通常是配置文件的第一个区块，用来定义启动顺序和依赖关系，主要字段：
    • Description：简短描述。
    • Documentation：文档地址。
    • Requires：当前 Unit 依赖的其他 Unit，如果它们没有运行，当前 Unit 会启动失败。
    • Before：如果该字段指定的 Unit 也要启动，那么必须在当前 Unit 之后启动。
    • After：如果该字段指定的 Unit 也要启动，那么必须在当前 Unit 之前启动。
  2. [Service] 区块通常只有 Service 类型的 Unit 才有，定义了如何启动当前服务，主要字段：
    • Type：定义启动时的进程行为，它有以下几种值：
      1. simple 表示执行 ExecStart 指定的命令来启动主进程；
      2. forking 表示以 fork 方式从父进程创建子进程，创建后父进程会立即退出；
      3. idle 类似 simple，但是要等到其他任务都执行完，才会启动该服务。一种使用场合是为让该服务的输出不与其他服务的输出相混合。
    • ExecStart ：启动当前服务的命令。
    • ExecStartPre：启动当前服务之前执行的命令。
    • ExecStartPost：启动当前服务之后执行的命令。
    • ExecReload：重启当前服务时执行的命令。
    • ExecStop：停止当前服务时执行的命令。
    • Environment：指定环境变量。
  3. [Install] 通常是配置文件的最后一个区块，用来定义如何启动，以及是否开机启动。主要字段：
    • WantedBy：它的值是一个或多个 Target。当前 Unit 激活 (enable) 时符号链接会放入 /etc/systemd/system 目录下面以 Target 名 + .wants 后缀构成的子目录中。例如，WantedBy=multi-user.target 的含义是，该 Unit 所在的 target 是 multi-user.target。
    • RequiredBy：它的值是一个或多个 Target，当前 Unit 激活时，符号链接会放入 /etc/systemd/system 目录下面以 Target 名 + .required 后缀构成的子目录中。
    • Alias：当前 Unit 可用于启动的别名。
    • Also：当前 Unit 激活 (enable) 时，会被同时激活的其他 Unit。
